1.1. Create a Basic Virtual Environment

Virtual environments have many benefits, not the least of which involves collaborative projects, where you might want to share the exact minimum requirements to make a project run.

1.1.1. Basic Environment

  • create new virtual environment (venv) with name myenv
$ conda create --name myenv
  • activate created venv
$ conda activate myenv

1.1.2. Specify Dependencies Upon Creation

  • create new venv with name myenv with specified Python version
$ conda create --name myenv python=3.10

1.1.3. Create venv in Project Directory

  • create venv in subdirectory, env
$ conda create --prefix ./env
  • activate new venv in created subdirectory
    • navigate to your project directory
$ cd project-directory
  • activate env
$ conda activate ./env
  • NOTE: this venv's location will not be known by conda, so you always have to manually specify path to directory when activating
    • upsides to this method include not having to worry about unique environment names and redistribution of exact environment used in project

1.1.3.1. Specify More Complicated Dependencies

  • create venv in subdirectory, env with specified python and package versions
    • can also specify packages with conda create --name command
$ conda create --prefix ./env python=3.10 jupyterlab=3.2 matplotlib=3.5 numpy=1.21
  • NOTE: After creating a venv using --prefix, you'll notice this environment is shown on the command line with a much longer name (full path to venv)
    • Use the following command if you wish to change this behavior:
$ conda config --set env_prompt "<{name}>"

  • where ""{name}"" is the syntax to capture the name of the directory you put the environment in

    • NOTE "{name}" is a string literal, and name should not be changed. The change is automatic, where the {} are a templating syntax, and name refers to name as referenced by the conda environment

  • Below, we begin to explore venv specification files, which provide a much more robust means of venv dependency specification, generation, and distribution.

1.2. YAML Files

When using conda as a package/environment handler, YAML files offer the creater the most flexible, easy to share, and reproducible means of specifying dependencies. Additionally, the recipient is given an easy to understand and robust means through which to initialize an environment having all necessary dependencies to get started with their project. This is my go-to method for dependency sharing.

1.2.1. Generate YAML File

  • write new environment.yml file with current venv packages specified
$ conda env export > environment.yml
  • create new venv from environment.yml file (preferred by conda)
$ conda env create -f environment.yml

1.2.2. Cross-Platform YAML Specification

  • write with only manually install packages (using --from-history tag)
    • good for cross-platform environment sharing
$ conda env export --from-history > environment.yml

1.2.3. YAML to Update Existing venv

  • update venv in subdirectory, env, using environment.yml file
    • this can be after manually deleting no longer used packages, for example
conda env update --prefix ./env --file environment.yml  --prune
  • NOTE: After creating a venv using --prefix, you'll notice this environment is shown on the command line with a much longer name (full path to venv)
    • Use the following command if you wish to change this behavior:
$ conda config --set env_prompt "<{name}>"
  • where ""{name}"" is the syntax to capture the name of the directory you put the environment in
    • NOTE "{name}" is a string literal, and name should not be changed. The change is automatic, where the {} are a templating syntax, and name refers to name as referenced by the conda environment

1.3. TXT File Specification

Sometimes, your recipient may not be versed in conda, and instead prefer something like pip for package management.

1.3.1. Write TXT Specification

  • write installed packages from activated environment to spec-file.txt
$ conda list --explicit --md5 > spec-file.txt

1.3.2. Create venv from TXT Specification

  • create new environment from spec-file.txt with name myenv
$ conda create --name myenv --file spec-file.txt

1.4. Extras

Cloning, deleting, and specifying environment variables using conda & YAML

1.4.1. Clone an Environment

  • clone existing venv called myenv into another venv called myclone 
$ conda create --name myclone --clone myenv

1.4.2. Delete Existing Environment

  • delete a venv
$ conda env remove --name myenv

1.4.3. Environment Variables

Sometimes it's convenient to namespace your environment variables. Just as packages can be installed in a venv, so too can venv environment variables be set.

1.4.3.1. Use Environment Variables in Virtual Environment

  • set environment variable in venv (disappears after deactivating venv)
$ conda env config vars set my_var=value
  • print to stdout all venv environment variables
$ conda env config vars list
  • unset venv environment variable
$ conda env config vars unset my_var -n test-env

1.4.3.2. Environment Variables in YAML Specification

In the same way an environment may be duplicated by users by reading a YAML file, so too can the environment variables be specified to automate distribution of packages and necessary namespaced environment variables.

  • set environment variables in .yml file, such as the environment.yml above
variables:
  VAR1: valueA
  VAR2: valueB

1.4.3.3. Target Deployment Machines

One good reason to make environment variables available every time an environment is made might be for use in a target deployment machine or another containerized system (Docker, for instance).

  • Set environment variables automatically upon venv activation.
    • First, navigate to directory pointed to by environment variable, CONDA_PREFIX:
$ cd %CONDA_PREFIX%
  • Now, we need two directories
    • 1 for our activate script
    • 1 for our deactivate script
$ mkdir .\etc\conda\activate.d
$ mkdir .\etc\conda\deactivate.d
  • Now, lets create the scipts:
$ type NUL > .\etc\conda\activate.d\env_vars.bat
$ type NUL > .\etc\conda\deactivate.d\env_vars.bat
  • Finally, we can write the script syntax.
  • contents of activate.d\env_vars.bat
set MY_KEY='secret-key-value'
set MY_FILE=C:\path\to\my\file
  • contents of deactivate.d\env_vars.bat
set MY_KEY=
set MY_FILE=

1.4.3.4. Use Local Package Repository

  • index a local package repo
$ conda search -c file:///opt/channel/ --override-channels

1.5. Generated Environment File Syntax

1.5.1. YAML File Syntax

  • specify packages and venv name, myenv, in .yml file manually
    • this is what is written using conda env export --from-history > environment.yml command above
# run this: 
# >>> conda env create -f environment.yml
# to create your environment named 'test' (you can change environment name below, if you like)
name: test
channels:
  - defaults
  - conda-forge
dependencies:
  - python=3.10
  - pysimplegui
  - ipykernel
  - matplotlib
prefix: C:\Users\tnlma\Anaconda3\envs\test
  • NOTE: prefix: C:\Users\tnlma\Anaconda3\envs\test is automatically created when creating YAML file, as is name: test at the top
